# Understanding Vulnerabilities Series
**By Stephen Reilly**

This is a series of blog posts in which I explore IT vulnerabilities and their mitigation, divided into eighteen chapters:

| Post | Summary |
|------|---------|
| [Vulnerability in Cyber Security](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/0-Vulnerability_in_Cyber_Security.md) | Cybersecurity vulnerabilities are gaps between system expectations and reality that have evolved from isolated technical glitches into critical business risks driven by interconnectivity across software, hardware, network, and human vectors. |
| [Analytic Tools](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/1-Analytic_Tools.md) | Static Analysis (SAST) and Dynamic Analysis (DAST) tools work in tandem to detect vulnerabilities early in the development lifecycle, transforming security from a reactive bottleneck into a proactive quality pillar while evolving toward AI-enhanced and CI/CD-integrated solutions. |
| [Injection Prevention](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/2-Injection_Prevention.md) | Defense-in-depth strategies for preventing injection attacks including SQL, XSS, and command injection through parameterized queries, input validation, context-aware encoding, and layered testing while emphasizing that inconsistent implementation—not lack of solutions—is why these vulnerabilities persist. |
| [Mitigating CSRF](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/3-Mitigating_CSRF.md) | Cross-Site Request Forgery (CSRF) attacks trick authenticated users into performing unintended actions. Details defense strategies including anti-CSRF tokens, SameSite cookie attributes, header validation, and framework-specific protections to prevent state-changing requests from unauthorized origins. |
| [Patching](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/4-Patching.md) | Patching is the critical act of sealing known cracks in your software, rather than cosmetic work, emphasizing that unpatched systems with known vulnerabilities are essentially open invitations to attackers and must be integrated into a broader defense-in-depth strategy through asset prioritization and continuous improvement. |
| [The Purpose of CVEs](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/5-The_Purpose_of_CVEs.md) | CVEs function as a global passport database for software vulnerabilities, assigning unique identifiers that transform fragmented security flaw reporting into coordinated, actionable intelligence across the entire cybersecurity industry. |
| [Severity in CVE](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/6-Severity_in_CVE.md) | CVE severity scoring provides a theoretical vulnerability baseline, but real-world prioritization requires weighing contextual factors like exploit availability, asset criticality, and business impact since severity dictates urgency while context dictates priority. |
| [The CVE List](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/7-The_CVE_List.md) | The CVE List, managed by MITRE and delegated to a global network of CVE Numbering Authorities (CNAs), provides a structured workflow where trusted organizations reserve, coordinate, enrich, and publish unique vulnerability identifiers to ensure standardized tracking across the security ecosystem. |
| [CVE and CVSS Scores](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/8-CVE_and_CVSS_Scores.md) | CVE identifiers and CVSS severity scores function best as foundational inputs within a broader risk-based prioritization framework that adjusts for environmental context, exploit probability (EPSS), and known exploitation (KEV) rather than serving as the sole decision-making metric. |
| [Calculating CVSS](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/9-Calculating_CVSS.md) | This post demonstrates how to calculate CVSS v3.1 scores by assigning specific Base Metric values (like Attack Vector, Complexity, and Impact) to a vulnerability scenario—such as an unauthenticated RCE—to derive a numerical severity rating that can be further refined with Temporal and Environmental adjustments for real-world risk assessment. |
| [The Difference Between CWE & CVE](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/10-The_Difference_Between_CWE_and_CVE.md) | CWE provides a general taxonomy of software weakness types (the "root cause catalog") while CVE documents specific, publicly-known vulnerabilities in particular products (the "incident report log"), working together to enable both proactive prevention through secure development and reactive remediation through standardized tracking. |
| [Role of CWE](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/11-Role_of_CWE.md) | The CWE serves as a standardized "dictionary" of software weaknesses that enables cybersecurity engineers to establish a common language, automate and standardize testing tools, map specific flaws to actionable coding fixes for developers, and transform abstract security risks into measurable engineering metrics for proactive defense. |
| [Common CWEs & Their Impact](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/12-Common_CWEs_and_Their_Impact.md) | SQL Injection (CWE-89), Cross-Site Scripting (CWE-79), Path Traversal (CWE-22), Out-of-Bounds Write (CWE-787), Untrusted Deserialization (CWE-502), and CSRF (CWE-352) are top repeat offenders that cause critical impacts ranging from data theft and privilege escalation to remote code execution. |
| [CWE Taxonomy](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/13-CWE_Taxonomy.md) | The CWE taxonomy functions as a standardized dictionary of approximately 900+ software weakness types that enables consistent vulnerability identification across teams and tools, facilitates pattern recognition for systemic process gaps, integrates with CVSS for risk prioritization, aligns security work with compliance frameworks like OWASP and PCI-DSS, and transforms vulnerability management from reactive fire-fighting into a strategic, measurable engineering program. |
| [The Relationship Between CWE, CVE, & CVSS](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/14-The_Relationship_Between_CWE_CVE_and_CVSS.md) | CWE (weakness type), CVE (specific instance), and CVSS (severity score) form an interconnected vulnerability management chain where CWE identifies the root cause, CVE catalogs the specific vulnerability, and CVSS quantifies impact—together transforming raw security data into actionable intelligence for triage, prioritization, and long-term architectural hardening. |
| [Understanding a Vulnerability](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/15-Understanding_a_Vulnerability.md) | By dissecting the "PrintNightmare" (CVE-2021-34527) flaw, we illustrates that understanding a vulnerability requires analyzing its specific technical mechanism (improper access control in the Print Spooler), quantifying its severity via CVSS scores and real-world exploitability, and implementing a layered defense strategy that includes immediate patching, service disabling, and registry hardening to mitigate the risk of system-wide compromise. |
| [Analyzing Vulnerability Trends](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/16-Analyzing_Vulnerability_Trends.md) | The surge in Linux Kernel vulnerabilities reported in 2026 isn't necessarily due to worse code, but rather a "perfect storm" of AI-driven discovery tools exposing technical debt faster than ever while simultaneously introducing new flaws via LLM-assisted development and automated reporting pipelines. |
| [Identify CWEs](https://github.com/sreilly1977/dlh-cyber_security/blob/main/understanding_vulnerabilities/17-Identify_CWEs.md) | We use a Python SQL injection example to demonstrate how identifying the specific Common Weakness Enumeration (CWE-89) clarifies the attack vectors and explain why separating code from data is critical. |
