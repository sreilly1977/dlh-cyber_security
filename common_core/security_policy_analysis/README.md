# Security Policy Analysis Series
**By Stephen Reilly**

This is a series of blog posts in which I analyse and develop security policies:

| Post | Summary |
|------|---------|
| [Policy Analysis](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/0-Policy_Analysis.md) | This security policy analysis document critiques an organizational information security policy, then provides a comprehensive rewritten version with proper governance structure, roles, monitoring, and compliance mechanisms. |
| [Acceptable Use Policy](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/1-Acceptable_Use_Policy.md) | This document addresses a fictional company's recent security incidents by outlining the specific risks of unauthorized software and data mishandling, then provides a comprehensive rewritten Acceptable Use Policy that establishes clear rules for internet browsing, email usage, and software installation alongside explicit monitoring and enforcement procedures. |
| [Password Policy](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/2-Password_Policy.md) | This document critiques a fictional bank's original password policy for lacking specificity and modern security standards, then presents a comprehensive rewritten version that enforces strict complexity rules, mandates multi-factor authentication with tiered approval levels, requires secure credential storage via hashing and salting, and defines clear enforcement consequences for violations. |
| [Incident Response Policy](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/3-Incident_Response_Policy.md) | This document critiques a fictional manufacturing company's original incident response policy for lacking structure and actionable procedures, then provides a comprehensive rewritten version that defines a clear Incident Response Team structure with specific roles (including OT-specific considerations), outlines standardized reporting and assessment protocols, and details step-by-step response procedures aligned with the NIST lifecycle covering detection, containment, eradication, and recovery. |
| [Data Classification Policy](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/4-Data_Classification_Policy.md) | This document critiques a healthcare organization's original data classification policy for failing to meet HIPAA/GDPR requirements and lack specific handling rules, then presents a comprehensive rewritten version that defines four distinct classification levels (Public, Internal, Confidential, Restricted) with mandatory labeling, encryption, access control, storage, transmission, and disposal procedures tailored to each sensitivity level. |
| [Policy Review Exercise](https://github.com/sreilly1977/dlh-cyber_security/blob/main/security_policy_analysis/5-Policy_Review_Exercise.md) | This document conducts a comprehensive gap analysis for RetailMax Corporation against PCI-DSS, ISO 27001, and SOC 2 frameworks, revealing that the company's fragmented and aging policies place it at a "Level 1" maturity stage with only 5–10% compliance coverage, and proposes a prioritized 12-month remediation plan to address critical missing policies and establish an auditable security program. |
